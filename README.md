# Loop Station

Loop Station is an early tmux-first Codex orchestration harness for building custom repeatable agent loops.

It ships a project-local Codex skill plus a project-local `.loop-station` runtime that coordinates visible Codex panes, dispatches human-readable task envelopes, watches artifacts and transcripts, and keeps loop state explicit. The project is intended for local inspection and workflow development, not as a polished production orchestrator. Use it to shape your own workflow around Codex panes: evaluation loops, recovery loops, installation loops, or multi-step automation pipelines.

The current bundled template is a generic kernel with typed built-in profiles. The default public path is preset-first: setup compiles interview answers into a profile such as `evaluation-loop`, `recovery-loop`, `installation-loop`, or `action-pipeline`. Freeform legacy station authoring remains available as an advanced compatibility path.

## Why Loop Station

Loop Station was inspired by excellent existing orchestrator and integration skills. Those tools show how useful agent coordination can be when work needs to be split across roles, tools, and feedback loops.

This project explores a smaller and more explicit coordination model: mailbox-style task handoff plus artifact-based evidence. Instead of assuming every pane or agent shares one long context, Loop Station treats each step as a bounded unit of work. A pane receives a readable task envelope, writes artifacts, and lets the station decide the next transition from files, manifests, transcripts, and verifier results.

There are several useful models for agent orchestration. Some share context directly. Some delegate to lower-level CLIs or isolated workers. Loop Station is aimed at cases where the developer wants to own the loop shape: which panes exist, what each pane sees, what artifacts count as evidence, and when a repair, judge, provider, or next action is allowed to run.

The bet behind this project is pragmatic: as model and tool providers keep changing context limits, pricing, isolation rules, and execution surfaces, durable workflows often need a small local harness that developers can inspect and customize.

## What it is for

- Build a project-local tmux + Codex loop around a workflow you define.
- Run skill benchmarks with sandbox/consumer cases and artifact-based verification.
- Run skill improvement loops with sandbox, judge, and provider roles.
- Orchestrate multi-step automation pipelines such as `action 1 -> action 2 -> action 3`.
- Keep each pane visible, inspectable, and bounded by explicit task envelopes.
- Use mailbox-style handoff instead of assuming every role shares one long context.
- Evaluate progress through artifacts, manifests, transcripts, and verifiers instead of chat-only self-report.
- Gate repair, provider handoff, judge review, or the next action on explicit evidence.

## Quick Start

```bash
node ./bin/loop-station install-skill --project ./my-project
node ./bin/loop-station setup --project ./my-project --spec ./my-setup.json
cd ./my-project
node ./.loop-station/bin/station validate --skip-tools
```

`install-skill` preserves an existing project-local `.codex/skills/loop-station` by default. `setup` requires that local skill install and generates `.loop-station`. For a destructive clean test refresh from the canonical template, use `node ./bin/loop-station install --replace --project ./my-project`.

Codex Install Mode should normally install the project-local skill first, then interview the user before final station configuration. Codex Run Mode should execute station commands for the user. Direct shell use is for debugging and review.

When Codex cannot provide a real tmux client context, Loop Station launches the runtime in a separate attached Terminal.app session. The current Codex conversation is the trigger/control session; live runner work belongs in the attached runtime panes after `station start`, not in the trigger session beforehand. The public runtime surface is always visible; `station start` does not support detached/background-only startup.

Verification is split deliberately:

- `npm test` covers deterministic runtime and contract behavior only.
- Live tmux/Codex runs are operator-driven local checks and may depend on current Codex CLI behavior, model latency, and pane readiness.

## Project Layout

Loop Station is expected to be installed as a Codex skill in the user project:

```text
user-project/
  .codex/skills/loop-station/
  .loop-station/
```

- `.codex/skills/loop-station/` is the installed Codex skill bundle.
- `.loop-station/` owns station config, cases, runs, and the vendored harness generated by setup.

The installed project-local `.codex/skills/loop-station/` bundle is a read-only payload in consumer projects. Do not treat `.codex/skills/loop-station/assets/harness-template/src/**` or `test/**` as editable project workspace. Consumer-side mutation belongs in `.loop-station/**`, project-local wrapper skills outside the installed loop-station bundle, or ordinary project files.

## Install vs Setup

There are two distinct project-local phases:

1. Install the skill into `.codex/skills/loop-station`
2. Run setup to generate `.loop-station`

`loop-station install-skill` must not create `.loop-station`.
`loop-station setup` must not auto-install the skill.

`loop-station install` remains a low-level harness/template install surface for debugging and review; it is not the default Codex skill install path.

## Target Skill Layout

Loop Station checks configured target skills before dispatch. Preset profiles generate typed role sets and phase graphs automatically; single-skill profiles use `targetSkillName`, while multi-skill pipelines use `targetSkills` plus `stageContracts` so the runner sees the public skill entries, stage inputs, required outputs, and evidence expectations.

The default target skill path is `<consumerRoot>/.codex/skills/<skill-name>`, and explicit install paths can still be configured. Install Mode should prefer copying user-level or explicitly sourced skills into the project-local consumer `.codex/skills` directory for reproducible runs.

The folder must contain exactly one `SKILL.md`/`skill.md` file with YAML front matter containing `name` and `description`. The manifest `name` must match `targetSkillName` without the leading `$`.

## Evaluation Gate

Runner artifacts are pass candidates, not final success. A case advances only after the typed `judgment` role writes `eval-report.md` and `eval-verdict.json` with a pass verdict. In `recovery-loop`, provisional passes must survive challenge review before the case can pass, and failed attempts route through provider fix plus deploy verification before rerun.

## Safety Model

Loop Station does not directly patch provider source, case inputs, or generated consumer artifacts. Provider changes belong to a provider-rooted Codex pane. Raw station control JSON is kept out of model-visible prompts by default.

See `docs/security-boundaries.md` and `docs/runtime-contract.md`.

## Public Examples

The bundled example is a public guide fixture. `echo-skill-loop` is a compact preset profile example that validates station config, case manifests, target skill install paths, mailbox activation semantics, and public path hygiene without private provider repos. The deterministic test suite validates its shape; the operator entrypoint can be used for a real visible tmux/Codex runtime check during local development.

```bash
STATION_CONFIG=examples/echo-skill-loop/station.json node skills/loop-station/assets/harness-template/bin/station validate --json --skip-tools
```

The repo also ships operator-friendly entrypoints that reset prior runs and start the visible tmux runtime:

```bash
cd examples/echo-skill-loop
./reset-and-run.sh
```

See `docs/review-package.md` for review commands and known risks.
